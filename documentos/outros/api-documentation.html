3.6 WebAPI e Endpoints

A API do projeto Red Bull 24 Horas foi organizada por domínios de uso da aplicação: competições, equipes, atletas, administradores, checkpoints, ranking, exportação e relatórios. Os endpoints listados como implementados correspondem ao conjunto de rotas registrado no backend atual. Autenticação, OCR, inconsistências e relatórios permanecem documentados como funcionalidades planejadas pelo WAD.

Media type: application/json; charset=utf-8 Formato de erro: JSON Implemented Planned

Convenções globais

  • Versionamento: a API não utiliza prefixo de versão (/v1/) nesta fase. Caso ocorram breaking changes em futuras sprints, será introduzido versionamento explícito por path.
  • Convenção REST: recursos são organizados hierarquicamente (ex.: /competitions/:competicaoId/teams/:teamId/athletes) e os métodos HTTP seguem semântica padrão — POST cria, GET consulta, PUT atualiza por substituição, PATCH atualiza parcialmente, DELETE remove.
  • Identificadores numéricos: todos os parâmetros de rota (:id, :competicaoId, :teamId, :athleteId, :corredorId, :extractionId) devem ser inteiros positivos. Valores inválidos retornam 400 Bad Request.
  • Content-Type: requisições com corpo devem usar Content-Type: application/json com codificação UTF-8. Respostas seguem application/json; charset=utf-8, suportando acentuação completa do português.
  • Autenticação: nenhuma rota exige autenticação na Sprint 3. A camada de autenticação por token será introduzida na Sprint 5 e passará a proteger as rotas administrativas (/administradores, /checkpoints, encerramento de competições, etc.). As rotas públicas via UUID (ranking público da equipe) permanecerão sem autenticação após a entrega.
  • Paginação: não implementada nesta fase. Endpoints GET de listagem retornam todos os registros relevantes em um único array. A decisão é compatível com o volume esperado da competição (até 16 equipes × 16 atletas = 256 atletas e algumas centenas de checkpoints por evento de 24 horas) e será reavaliada caso esse volume aumente significativamente.
  • Filtros via query string: não implementados nesta fase. A filtragem por contexto (competição, equipe, corredor) é feita via aninhamento do path, que é a forma idiomática RESTful para recursos hierárquicos. A ordem do aninhamento é fixa e segue a hierarquia natural do domínio: competição → equipe → atleta → checkpoint.
  • Atletas e corredores: os termos são usados de forma intercambiável neste documento. Atleta é o nome canônico da entidade no código e nos endpoints aninhados (/competitions/:competicaoId/teams/:teamId/athletes); corredor permanece como termo legado em endpoints históricos (/corredores/:corredorId/checkpoints) e em valores de campo (status: "corredor") por compatibilidade com a modelagem original do banco.
  • Idempotência: GET, PUT e DELETE são idempotentes; POST e PATCH não são. Clientes devem aplicar política de retry consciente desse comportamento.
  • Rastreabilidade: cada endpoint está vinculado a um ou mais Requisitos Funcionais (RF) e Regras de Negócio (RN) listados na coluna RF/RN das tabelas. A correspondência completa com User Stories está consolidada na seção 3.9 (Matriz de Rastreabilidade) do WAD.
  • Tratamento global de erros: todos os erros são serializados pelo middleware errorHandler a partir das classes em src/errors/AppError.ts (ValidationError → 400, NotFoundError → 404, ConflictError → 409, UnprocessableError → 422, fallback → 500).

Authentication

Nesta fase nenhuma rota está protegida. A partir da Sprint 5, as rotas administrativas passarão a exigir um token Bearer; o formato esperado da requisição autenticada é: 

Authorization: Bearer <token>
Content-Type: application/json

Resposta padrão de erro

{
  "message": "Descrição do erro"
}

Códigos de saída HTTP

Código Uso esperado Formato de resposta
200 OK Consulta, listagem, atualização ou operação concluída com retorno no corpo da resposta. JSON com objeto ou lista de objetos.
201 Created Recurso criado com sucesso por uma requisição POST. JSON com o recurso criado e seu identificador.
204 No Content Exclusão ou operação concluída sem corpo de resposta. Sem body.
400 Bad Request Payload inválido, parâmetro ausente ou parâmetro numérico em formato incorreto. { "message": "Descrição do erro" }
401 Unauthorized Token ausente, inválido ou expirado quando a autenticação estiver habilitada. { "message": "Token inválido ou expirado" }
403 Forbidden Usuário autenticado mas sem permissão para a operação solicitada. Reservado para a Sprint 5, quando a camada de autorização por perfil for introduzida. { "message": "Acesso negado" }
404 Not Found Recurso solicitado não encontrado. { "message": "Recurso não encontrado" }
409 Conflict Conflito de regra de negócio, como duplicidade de dado único. { "message": "Conflito ao processar recurso" }
422 Unprocessable Entity Dados sintaticamente válidos, mas inconsistentes com a regra de negócio. { "message": "Dados inconsistentes" }
500 Internal Server Error Falha inesperada na API ou na integração com o banco. { "message": "Erro interno do servidor" }

3.6.6 Ranking Routes

As rotas de ranking retornam leituras agregadas por equipes e atletas, permitindo acompanhar posições, distância acumulada e métricas consolidadas da competição.

Método Endpoint Status Descrição RF/RN
GET /competitions/:competitionId/ranking/teams Implemented Retorna ranking de equipes. RF010, RN09, RN11
GET /competitions/:competitionId/ranking/athletes Implemented Retorna ranking de atletas. RF015, RN09, RN11

Response - 200 OK

{
  "competicao_id": 1,
  "atualizado_em": "2026-05-28T10:00:00.000Z",
  "ranking": [
    {
      "posicao": 1,
      "equipe_id": 1,
      "equipe_nome": "Equipe Alpha",
      "distancia_total_km": 128.4,
      "pace_medio": "5:12/km",
      "diferenca_para_lider_km": 0
    }
  ]
}

Campos do response

  • competicao_id (number): identificador da competição.
  • atualizado_em (string ISO 8601): timestamp da última atualização do ranking.
  • ranking (array): lista ordenada por posição (ascendente).
  • ranking[].posicao (number): posição da equipe ou atleta no ranking.
  • ranking[].equipe_id / ranking[].atleta_id (number): identificador do recurso, conforme a rota consultada.
  • ranking[].equipe_nome / ranking[].atleta_nome (string): nome do recurso.
  • ranking[].distancia_total_km (number): distância acumulada em quilômetros, com até três casas decimais.
  • ranking[].pace_medio (string no formato M:SS/km ou MM:SS/km): pace médio acumulado.
  • ranking[].diferenca_para_lider_km (number): diferença em quilômetros para a 1ª posição (0 para o líder).

Saídas da requisição

Operação Sucesso Erros esperados Observação
GET /competitions/:competitionId/ranking/teams 200 OK 400, 404, 500 Retorna ranking consolidado de equipes.
GET /competitions/:competitionId/ranking/athletes 200 OK 400, 404, 500 Retorna ranking consolidado de atletas.

3.6.11 TV Panel Routes

A rota do Painel TV consolida em uma única chamada pública as métricas agregadas da competição em andamento — pace médio geral, tempo decorrido, quilometragem total e equipes em destaque — destinadas à exibição contínua em monitores durante o evento, sem necessidade de autenticação ou interação do operador. Atende à US19 (#519).

Método Endpoint Status Descrição RF/RN
GET /public/competitions/:competitionId/tv-panel/metrics Implemented Retorna métricas consolidadas da competição para o Painel TV. RF010, RN09, RN11

Rota pública, sem autenticação, seguindo o mesmo padrão de acesso adotado em GET /public/team/:uuid (US12).

Query parameters

  • topN (number, opcional): quantidade de equipes retornadas em metrics.top_teams. Inteiro entre 1 e 10; o valor padrão é 3. Valores fora do intervalo resultam em 400 Bad Request.

Response - 200 OK

{
  "id_competition": 5,
  "competition_name": "Red Bull 24h Sao Paulo",
  "competition_status": "in_progress",
  "metrics": {
    "average_pace_overall": "5:00",
    "average_pace_overall_seconds": 300,
    "elapsed_time": "03:42:15",
    "elapsed_time_seconds": 13335,
    "total_distance_km": 487.5,
    "top_teams": [
      {
        "position": 1,
        "id_team": 10,
        "team_name": "Equipe Alpha",
        "total_distance_km": 142.3,
        "average_pace": "4:55",
        "average_pace_seconds": 295
      }
    ]
  },
  "generated_at": "2026-06-22T11:00:00.000Z"
}

Campos do response

  • id_competition (number): identificador da competição consultada.
  • competition_name (string): nome da competição.
  • competition_status (string): valor entre not_started, in_progress e closed.
  • metrics.average_pace_overall (string M:SS ou null): pace médio coletivo da competição, formatado em minutos e segundos por quilômetro. Calculado como média ponderada do average_pace_seconds de cada corredor pela respectiva quilometragem acumulada, o que equivale a Σ tempo / Σ km. Retorna null quando nenhum corredor da competição possui pace válido.
  • metrics.average_pace_overall_seconds (number ou null): a mesma métrica em segundos por quilômetro, sem arredondamento de formatação.
  • metrics.elapsed_time (string HH:MM:SS ou null): tempo decorrido da prova desde o seu início efetivo.
  • metrics.elapsed_time_seconds (number ou null): o mesmo valor expresso em segundos.
  • metrics.total_distance_km (number): quilometragem total acumulada por todas as equipes da competição.
  • metrics.top_teams (array): equipes em destaque, ordenadas pelo ranking geral de equipes (maior total_distance_km, com average_pace_seconds como critério de desempate). O tamanho máximo do array é controlado pelo parâmetro topN.
  • metrics.top_teams[].position (number): posição da equipe no ranking geral da competição.
  • metrics.top_teams[].id_team (number): identificador da equipe.
  • metrics.top_teams[].team_name (string): nome da equipe.
  • metrics.top_teams[].total_distance_km (number): quilometragem acumulada da equipe.
  • metrics.top_teams[].average_pace (string M:SS ou null): pace médio da equipe formatado.
  • metrics.top_teams[].average_pace_seconds (number ou null): pace médio da equipe em segundos por quilômetro.
  • generated_at (string ISO 8601): timestamp de geração do payload.

Nota técnica sobre elapsed_time. O esquema atual da tabela competition (vide migration 0001_create_competition.sql) não possui colunas formais para o instante de início e término da prova. Enquanto a feature de iniciar e finalizar competição não entra em produção, o serviço utiliza como proxy o created_at do checkpoint mais antigo da competição como início efetivo da prova, e o created_at do checkpoint mais recente como término quando competition_status é closed. Quando ainda não há checkpoints registrados, elapsed_time e elapsed_time_seconds retornam null. A substituição desse proxy pela leitura direta dos campos formais de início e término está registrada como Trabalho Futuro (seção 7 do WAD) e é a única função do tvPanelService afetada pela transição.

3.6.12 Compartilhamento / Share Routes

As rotas de compartilhamento permitem a geração de templates visuais no estilo Instagram com destaques da competição (maior KM, menor pace, desempenho por período) para divulgação em redes sociais. Os templates são renderizados no servidor (SSR) e podem ser baixados como imagem PNG no lado do cliente via html2canvas. Atende ao RF016 (#514, #515).
